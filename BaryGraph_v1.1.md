# BaryGraph
## A CM-Topology Knowledge Graph Architecture
**Version 1.1 · April 2026**

> **v1.1 change:** HierarchyLinks removed. MetaBary connections now handle
> all cross-level relationships. Every connection in the graph — horizontal
> or vertical — is a BaryEdge or MetaBary. The triad topology is universal.

---

## 1. Motivation

Standard knowledge graphs treat relationships as thin directed edges:
label + weight between two nodes. Standard vector databases treat
documents as points in embedding space, retrieved by cosine similarity.
Both approaches lose relational structure that is often the most valuable
part of a corpus.

Consider two papers: one reporting an unexplained spacecraft flyby anomaly
in planetary science, another reporting an anomalous orbital residual in
stellar dynamics. Neither cites the other. Their vectors are distant in
embedding space. Standard retrieval returns neither when querying the
other. Yet they describe the same underlying physical phenomenon at
different scales.

BaryGraph makes that connection first-class and queryable. The
relationship between the two papers is not a thin edge — it is a document
with its own vector, its own properties, and its own position in the
hierarchy. It can be retrieved directly, traversed, and used to discover
further bridges.

## 2. Core Concepts

### 2.1 The Triad as Atomic Unit

Every connection in BaryGraph is a triad of three entities:

```
CM₁  →  BaryEdge  →  CM₂
```

CM₁ and CM₂ are knowledge nodes (papers, concepts, domains, authors). The
BaryEdge is the relationship document sitting between them. It is not a
pointer — it is a first-class stored object with its own embedding vector,
its own properties, and its own identity. It can be retrieved, queried,
and promoted to a CM role in MetaBary connections.

### 2.2 Hierarchy Levels

All nodes and BaryEdges carry a `level` property (1–15). The level
determines the scale at which a connection operates:

| Level | Scale | Examples |
|---|---|---|
| 1–3 | Paradigm / Theory | Physics, Biology, Computer Science |
| 4–6 | Field / Sub-field | Astrophysics, Molecular Biology, NLP |
| 7–9 | Mechanism / Method | Orbital mechanics, CRISPR, Transformers |
| 10–12 | Topic cluster | Flyby anomalies, gene regulation, attention |
| 13–14 | Individual paper | Specific arXiv / PubMed documents |
| 15 | Entity / Claim | Named measurement, specific claim, data point |

**BaryEdges connect two nodes at the same level (horizontal).**
Cross-level relationships are handled by MetaBary connections (see §2.4).

### 2.3 The BaryEdge Vector

Every BaryEdge has an embedding vector computed as a weighted mixture of
its two CM vectors plus a relationship-type vector:

```
L|bary_vec = normalize( q·v(L|CM₁) + q·v(L|CM₂) + (1−q)·v(type) )
L+2|meta_bary = normalize( q·v(L|BE₁) + q·v(L|BE₂) + (1−q)·v(L+1|BE) )
```

where `q` is connection quality (0–1) and `v(type)` is the embedding of
a fixed natural-language description of the edge type and `L` is hierarchy level. A strong connection
(`q → 1`) produces a barycenter close to both CM vectors simultaneously.
A weak connection (`q → 0`) tilts the barycenter toward the semantic
space of the relationship type itself.

The BaryEdge also stores a `registry.summary` — a short natural-language
description of what the connection means, generated by an LLM at
ingestion time. This summary is embedded as a separate `summary_vector`
field, providing a second retrieval signal independent of the structural
`bary_vec`. When a query lands near the barycenter, it implicitly surfaces
both parent CMs.

### 2.4 MetaBary: BaryEdges as CMs — the Universal Connector

At level 8 and above, BaryEdges become eligible to act as CMs for new
connections. Two BaryEdges — at the **same or different levels** — can be
connected by a MetaBary:

```
BaryEdge_A (level Lₐ)  →  MetaBary  →  BaryEdge_B (level L_b)
```

MetaBary serves two distinct roles depending on the level relationship:

**Same-level MetaBary (Lₐ = L_b):** encodes cross-paradigm insight — the
relationship between relationships at the same abstraction scale.
Example: `"flyby anomaly ↔ Pioneer anomaly"` → MetaBary →
`"dark matter rotation curves ↔ MOND"` — both pairs describe unexplained
gravitational excess at system boundaries.

**Cross-level MetaBary (Lₐ ≠ L_b):** encodes hierarchical abstraction —
a specific relationship at one level is an *instance of* a broader
relationship at a higher level. This replaces HierarchyLinks entirely.
Example: a BaryEdge connecting two financial senses at L15 is connected
via MetaBary to a BaryEdge connecting two economic concept clusters at
L10. The MetaBary says: "this specific sense relationship is an instance
of this broader conceptual relationship."

**Why MetaBary is better than HierarchyLinks:**

HierarchyLinks were a special-purpose connector that broke the fundamental
rule: every connection is a triad. A HierarchyLink was just a pointer
with an `abstraction_loss` weight — semantically thin, not retrievable as
a document, and architecturally inconsistent. MetaBary preserves the
triad topology at every level and every direction. The graph has one
connection type, not two.

**Validity constraints:**

- *Cross-level MetaBary:* no common-ancestor requirement. The level
  difference itself establishes hierarchy direction (`level(cm1) <
  level(cm2)` means cm2 is more abstract). Direction is stored in
  `hierarchy_direction` field.
- *Same-level MetaBary:* requires a verified common ancestor in the
  MetaBary lineage to prevent spurious cross-paradigm connections.

**Isolated nodes:** a node with no BaryEdges at its level has no MetaBary
path upward. This is correct behavior — a concept with no relationships
has no place in the abstraction hierarchy.

## 3. Data Schema

All documents live in a single MongoDB collection. The `doc_type` field
distinguishes entity nodes from BaryEdges.

### 3.1 CM Node

| Field | Description |
|---|---|
| `_id` | ObjectId — primary key |
| `doc_type` | `"node"` |
| `node_type` | `"paper"` \| `"domain"` \| `"concept"` \| `"author"` \| `"institution"` \| `"paradigm"` |
| `level` | Integer 1–15 |
| `label` | Human-readable name or title |
| `vector` | Float array — embedding of label + abstract/description |
| `surface` | Integer — maximum BaryEdges this node can sustain |
| `rotation` | Float — update frequency |
| `properties` | Type-specific payload |
| `created_at` / `updated_at` | Dates |

### 3.2 BaryEdge

| Field | Description |
|---|---|
| `_id` | ObjectId — primary key |
| `doc_type` | `"baryedge"` |
| `cm1_id` | ObjectId → CM Node |
| `cm2_id` | ObjectId → CM Node |
| `level` | Must equal `cm1.level == cm2.level` |
| `edge_type` | `"cites"` \| `"extends"` \| `"contradicts"` \| `"same_phenomenon"` \| `"applies_to"` \| `"is_instance_of"` |
| `q` | Float 0–1 — connection quality |
| `strength` | Float — raw cosine similarity or citation count |
| `vector` | Float array — `bary_vec` (structural position signal) |
| `summary_vector` | Float array — embedding of `registry.summary` (semantic signal) |
| `registry` | `{ shared_concepts: [String], divergence: [String], summary: String }` |
| `is_metabary` | Boolean — true when both `cm1` and `cm2` are BaryEdges |
| `hierarchy_direction` | `"up"` \| `"lateral"` \| `null` — set when `is_metabary = true` |
| `common_ancestor_id` | ObjectId — required for same-level MetaBary only |
| `source` | `"ingested"` \| `"manual"` \| `"inferred"` |
| `confidence` | Float 0–1 |
| `created_at` / `updated_at` | Dates |

**`hierarchy_direction` values:**

- `"up"` — cross-level MetaBary; `cm1.level > cm2.level` (cm2 is more
  abstract). Encodes "cm1 relationship is an instance of cm2 relationship."
- `"lateral"` — same-level MetaBary. Encodes cross-paradigm connection.
- `null` — standard BaryEdge (not a MetaBary).

## 4. Ingestion Pipeline

### 4.1 Level 14–15: Base Nodes

Each document is embedded and stored as a CM Node at level 14–15. Named
entities, specific claims, and measurements become level-15 nodes.

### 4.2 BaryEdge Formation (Bottom-Up, Same Level)

For each level from 15 up to 8:

1. Compute cosine similarity across all node pairs at this level
2. For pairs above threshold (default 0.72), generate `registry.summary` via LLM
3. Compute `bary_vec = normalize(q·v(cm1) + q·v(cm2) + (1−q)·v(edge_type))`
4. Embed `registry.summary` → `summary_vector`
5. Store BaryEdge document

At level 15, for structured corpora (e.g. kaikki.org), seed BaryEdges
directly from explicit relation arrays rather than pairwise scan.

### 4.3 Domain Nodes (Level 10–6)

Domain nodes are created by clustering BaryEdges from the level below.
The domain node vector is the centroid of its children's BaryEdge vectors,
q-weighted.

### 4.4 MetaBary Formation

**Same-level MetaBary (lateral):**
- Eligible at level 8 and above
- Pairs of BaryEdges at the same level with cosine similarity above
  threshold (default 0.80)
- Requires common ancestor in MetaBary lineage
- `hierarchy_direction = "lateral"`

**Cross-level MetaBary (upward):**
- Eligible at any level combination where both BaryEdges exist
- Connects a BaryEdge at level Lₐ to a BaryEdge at level L_b where Lₐ > L_b
- No common-ancestor requirement
- `hierarchy_direction = "up"`, set `cm1` to the lower-level (more specific) BaryEdge
- Replaces HierarchyLinks: a cross-level MetaBary *is* the hierarchy

## 5. Query Patterns

### 5.1 Standard Retrieval

Vector search across all `doc_type` values simultaneously. Returns a mix
of nodes, BaryEdges, and MetaBarys ranked by relevance.

### 5.2 Relationship-Aware Retrieval

```js
db.barygraph.aggregate([
  { $vectorSearch: {
      index: 'barygraph_vector', path: 'vector',
      queryVector: embed('query text'),
      numCandidates: 200, limit: 20,
      filter: { doc_type: { $in: ['node', 'baryedge'] }, level: 15 }
  }},
  { $lookup: { from: 'barygraph', localField: 'cm1_id', foreignField: '_id', as: 'cm1' }},
  { $lookup: { from: 'barygraph', localField: 'cm2_id', foreignField: '_id', as: 'cm2' }}
])
```

### 5.3 Cross-Domain Bridge Query

```js
filter: { edge_type: 'same_phenomenon' }
```

Returns only BaryEdges bridging different domains. Each result implies
both parent nodes as context.

### 5.4 Hierarchy Traversal via MetaBary

```js
// From a BaryEdge, climb to more abstract MetaBary connections
db.barygraph.aggregate([
  { $match: { _id: baryEdgeId }},
  { $graphLookup: {
      from: 'barygraph',
      startWith: '$_id',
      connectFromField: '_id',
      connectToField: 'cm1_id',
      as: 'upward_chain',
      maxDepth: 8,
      restrictSearchWithMatch: {
        is_metabary: true,
        hierarchy_direction: 'up'
      }
  }}
])
```

Walking upward through cross-level MetaBary connections replaces the old
three-step HierarchyLink traversal. The chain is a direct `$graphLookup`
on `cm1_id` filtered by `hierarchy_direction: "up"`.

### 5.5 MetaBary Pattern Query

```js
filter: { is_metabary: true, hierarchy_direction: 'lateral' }
```

Returns same-level MetaBary connections — cross-paradigm bridges between
relationship families.

## 6. Key Architectural Rules

- **BaryEdges** connect two nodes at the **same level only** (horizontal).
- **MetaBary** connects two BaryEdges at **any level combination**
  (horizontal or vertical). This is the only cross-level connector.
- There are **no HierarchyLinks**. The triad topology is universal.
- A node with no BaryEdges at its level has no upward MetaBary path.
  This is correct: a concept with no relationships has no hierarchy
  position.
- `hierarchy_direction` is required on all MetaBary documents.
- Same-level MetaBary requires `common_ancestor_id`. Cross-level does not.
- `bary_vec` is always computed from CM vectors algebraically — never
  embedded directly. If a CM vector updates, its BaryEdge vectors must
  be recomputed.
- `summary_vector` and `bary_vec` are kept separate. The evaluation
  measures each signal independently before any merge decision.
- Connection quality `q` decays with staleness:
  `q(t) = q₀ · exp(−λ·Δt)`

## 7. Infrastructure

Single MongoDB collection. Atlas Vector Search index on the `vector`
field covers all `doc_type` values simultaneously. Standard Atlas queries
handle filtering, MetaBary traversal, and CM parent resolution.

```json
{
  "fields": [
    { "type": "vector", "path": "vector", "numDimensions": 1536, "similarity": "cosine" },
    { "type": "filter", "path": "doc_type" },
    { "type": "filter", "path": "level" },
    { "type": "filter", "path": "edge_type" },
    { "type": "filter", "path": "is_metabary" },
    { "type": "filter", "path": "hierarchy_direction" }
  ]
}
```

## 9. Open Questions

1. Optimal cosine threshold for cross-level MetaBary formation — same
   threshold as lateral, or lower (more permissive for hierarchy)?
2. Whether `summary_vector` should be a separate mongot index or a second
   vector path in the primary index.
3. Whether domain node vectors (centroids of child BaryEdge vectors)
   should include MetaBary vectors in the centroid computation.
4. Formal evaluation metric for cross-level MetaBary correctness — no
   standard ground truth exists for "this specific relationship is an
   instance of this broader relationship."
5. Whether the `common_ancestor_id` constraint for same-level MetaBary
   can be relaxed at levels 1–3 where paradigm-level connections are
   intentionally broad.
